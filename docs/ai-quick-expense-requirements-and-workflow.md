# Zero AI 快速记账需求与开发流程

## 背景

Zero 是一个开源 Android/iOS 记账应用。本次开发目标是在保持应用极简风格和现有手动记账流程稳定的前提下，增加一个“AI 快速记账”功能。

用户可以输入一句话或多行自然语言消费记录，应用调用用户自己配置的 OpenAI-compatible LLM API，将文本解析为待创建支出账单。解析结果必须先展示给用户确认，用户可修改、删除、补全后再批量写入当前账本。

目标兼容火山引擎 / Doubao 的 OpenAI 兼容接口。

## 核心需求

- 用户可在设置中填写：
  - API Base URL
  - API Key
  - Model Name
- 用户可输入一句或多行自然语言消费记录。
- 单条和批量输入共用同一个入口，不需要切换模式。
- 支持输入示例：
  - `中午麦当劳 32 元，用支付宝`
  - `昨晚打车 48，备注去机场`
  - `今天买咖啡 18.5`
  - 多行文本：

```text
昨天：
午饭 28
瑞幸 16
地铁 4

今天：
麦当劳 32
打车去机场 48
便利店 12.5
```

- LLM 负责把自然语言拆分和解析为多条待创建账单。
- 本地代码负责：
  - JSON 解析
  - 金额校验
  - 日期默认值补齐
  - 标题校验
  - 分类匹配
  - 合法/非法条目拆分
  - 批量创建账单
- 默认只支持支出，不扩展收入、转账、预算、财务建议、自动报表等功能。

## 重要约束

- 不修改数据库 schema，除非后续确认确有必要。
- 不破坏现有手动新增/编辑账单流程。
- 不引入复杂聊天界面。
- 不引入复杂多语言系统。
- 不自动迁移旧用户的英文默认分类。
- 不发送完整账本历史给 LLM。
- 不在用户确认前写入账单。
- 不配置 API 时，AI 入口必须提示先去设置。
- 在确认编译环境前，不运行 Android/iOS 构建。
- 在确认前，不运行 typecheck、lint、test。

## 隐私与安全策略

默认只发送给 LLM：

- 用户当前输入文本
- 当前日期时间
- 当前可用分类名称

默认不发送：

- 完整账本历史
- 用户姓名和邮箱
- 历史消费记录
- API Key

API Key 保存策略：

- 先使用项目现有 MMKV 本地保存。
- API Key 不写入 Redux。
- API Key 不写入数据库。
- API Key 不进入导入/导出文件。
- API Key 不出现在日志、错误提示、调试输出中。
- 设置页需要明确提示：API Key 仅保存在本地，但不是系统 Keychain / Keystore 级别加密存储。

## LLM 解析原则

- 使用 OpenAI-style Chat Completions 接口。
- 请求路径兼容两种输入：
  - Base URL，例如 `https://example.com/v1`
  - 完整 URL，例如 `https://example.com/v1/chat/completions`
- 非 streaming 请求。
- `temperature` 使用 `0`。
- LLM 只返回 JSON，不返回 Markdown 或解释。
- LLM 不应直接决定真实 `categoryId`。
- LLM 只返回：
  - `categoryName`
  - `categoryHint`
- 本地代码根据当前可用分类匹配真实 `categoryId`。
- 分类匹配失败时：
  - 优先使用默认分类“其他”。
  - 如果没有可用默认分类，则标记用户补充。
- 金额缺失时：
  - `amount` 必须为 `null`。
  - 本地标记为非法条目。
  - 不允许创建该条。

## 解析结果确认 UI

AI 快速记账页面保持极简：

- 顶部标题：AI 快速记账。
- 一个多行文本输入框。
- 一个“解析”按钮。
- 一个结果确认区域。

确认列表中每条账单支持：

- 修改标题
- 修改金额
- 修改日期
- 修改分类
- 修改备注
- 删除该条

创建规则：

- 用户点击“确认创建”后才写入数据库。
- 合法条目正常创建。
- 非法条目跳过。
- 创建完成后提示成功数量和跳过数量。

## 汉化策略

汉化拆分为三个阶段，不和 AI 功能强耦合：

1. AI 相关汉化
   - AI 快速记账页
   - AI 设置页
   - AI 入口提示
   - AI 错误提示

2. 核心记账流程汉化
   - 首页
   - 新增账单
   - 编辑账单
   - 分类
   - 统计
   - 设置
   - 表单占位符
   - 空状态
   - Dialog/Alert

3. 全量扫尾汉化
   - onboarding
   - 导入/导出
   - 债务相关页面
   - 底部 Sheet
   - Error Boundary
   - 其他用户可见文案

默认分类策略：

- 新安装用户默认支出分类改为中文。
- 旧用户英文默认分类迁移暂不自动执行。
- 旧数据迁移作为后续可选任务，需要单独确认。

## 开发实施顺序

优先服务层和纯函数，再接 UI：

1. AI 配置读写服务
2. Prompt 构造
3. OpenAI-compatible 请求封装
4. LLM JSON 解析
5. 分类匹配
6. 待创建账单校验
7. 批量条目合法/非法拆分
8. AI 设置页
9. AI 快速记账页
10. 导航入口
11. AI 相关汉化
12. 核心记账流程汉化
13. 全量汉化扫尾
14. 静态自查
15. 在确认环境后再执行 typecheck、lint、test 或移动端构建

## 验收标准

- 不配置 API 时，AI 入口提示先去设置。
- 配置火山引擎 / Doubao OpenAI-compatible API 后，可以解析自然语言。
- 输入 `中午麦当劳 32 元，用支付宝`，能得到一条待确认支出：
  - 金额：32
  - 标题：麦当劳
  - 分类：近似餐饮
  - 日期：今天
  - 备注：用支付宝
- 输入多行文本时，能解析为多条待确认支出。
- 金额缺失的条目不能创建。
- 分类匹配失败时使用“其他”或要求用户确认。
- 用户确认后才创建账单。
- 批量创建后显示成功数量和跳过数量。
- 现有手动记账功能不受影响。
- API Key 不出现在日志、Redux、数据库、导入导出文件中。
- 不执行未经确认的编译或构建命令。

## 本地 Git 管理建议

建议从现在开始把开发过程拆成多个本地提交：

1. 基准提交
   - 记录当前已完成的 AI 快速记账和汉化工作。
   - 作为后续回退点。

2. 验证提交
   - 在确认环境后，运行 typecheck、lint、test。
   - 根据结果修复类型和静态检查问题。

3. 运行验证提交
   - 在确认移动端环境后，再运行 Android/iOS。
   - 修复运行时问题。

4. 清理提交
   - 调整 UI 文案、交互细节和边界错误提示。

建议提交粒度：

- 一个提交只表达一个明确目的。
- 不把构建产物、缓存文件、API Key、日志文件加入 Git。
- 每次提交前查看 Source Control 面板，确认 staged 文件符合预期。
