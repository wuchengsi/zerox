# Zerox 性能优化原则

## 目标

Zerox 是轻量记账应用，日常交互应保持即时、安静、低阻力。性能优化优先围绕用户高频路径展开：

- 首页返回和切换月份
- 新增、编辑、删除账单后刷新
- 收入/支出流水滚动
- AI 快速记账输入和队列恢复
- 首次进入、导入数据、默认分类补齐

## 固定原则

1. 避免无意义的全量刷新
   - 页面重新聚焦时不要默认废弃缓存。
   - 只有新增、编辑、删除、导入、AI 入账等真实数据变化后才主动 invalidate。
   - 普通页面回退、Tab 切换应优先复用 Redux 缓存。

2. 列表必须按单条记录虚拟化
   - 长流水页面优先使用 `FlashList`。
   - 不要用 `ScrollView + map` 渲染大量账单。
   - 不要把一整天账单作为一个虚拟化 item 再在内部 map；日期标题和单条账单都应是列表 item。

3. 减少渲染阶段对象重建
   - 不在组件渲染中为整月数据批量 `{...item}`。
   - 可在 service/thunk 层补齐稳定字段，例如 `transactionType`。
   - 聚合、排序、分组使用 `useMemo`，依赖要尽量稳定。

4. 避免输入过程同步重写存储
   - 文本输入应先更新 React state。
   - 写 MMKV / 本地存储使用 debounce。
   - 页面退出、提交等关键时机再做一次兜底保存。

5. 数据库写入尽量批量化
   - 默认分类补齐、导入、迁移不应逐条开启 `database.write`。
   - 优先使用一次 `database.write` + `database.batch`。
   - 对用户可见的轻操作保持小事务即可。

6. 统计和聚合按风险逐步下沉
   - 小数据量可在组件中 memoize。
   - 数据量增长后，应迁移到 selector、service，必要时由数据库层聚合。
   - 避免多个页面同时重复计算同一批月度数据。

7. 缓存必须有明确失效点
   - 缓存不是为了永不更新，而是为了避免无原因重复更新。
   - 所有改变账单、收入、分类、导入数据的路径都必须显式刷新相关缓存。
   - 只读导航不应触发强制刷新。

## 本轮已应用的优化

- 首页回焦不再强制 invalidate 支出/收入缓存。
- 首页账单列表改为单条记录级别的 `FlashList` 虚拟化。
- 收入页流水从 `ScrollView + map` 改为 `FlashList`。
- AI 快速记账输入保存改为防抖写入 MMKV。
- 默认分类补齐改为批量创建。
- 支出/收入 service 层直接返回 `transactionType`，减少首页合并列表时的对象复制。

## 后续建议

- 报表页的大类/小类统计后续可下沉为 memoized selector。
- 导入大量数据时应进一步减少逐条写入和逐条 dispatch。

## 文档维护原则

- README 记录当前产品能力，不保留 fork 前已经失效的功能描述。
- 需求文档记录已经确认的行为边界，避免把旧方案当成当前实现。
- 性能文档只记录可复用原则和已经验证的优化，不记录临时调试细节。
